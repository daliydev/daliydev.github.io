브릿지 패턴은 기능과 구현에 대해 클래스를 분리하여 독립적으로 변경이 가능하고 확장이 가능하도록 합니다.
즉, 양쪽 클래스의 결합도가 낮아져서 독립적으로 변경과 확장이 가능한 상태.

예제) 접속할 OS에 로그인이 성공하면 정보를 출력.

public interface OS {
    void info();
}

public class Window implements OS {
    
    @Override
    public void info() {
        try {
            Runtime.getRuntime().exec("systeminfo");
        }catch (Exception e){
            e.printStackTrace();
        }
    }
}

public class Linux implements OS {
    
    @Override
    public void info() {
        try {
            Runtime.getRuntime().exec("free -m");
        }catch (Exception e){
            e.printStackTrace();
        }
    }
}

public class MacOS implements OS {
    
    @Override
    public void info() {
        try {
            Runtime.getRuntime().exec("vm_stat");
        }catch (Exception e){
            e.printStackTrace();
        }
    }
}

public abstract class User {
    protected OS os;
    
    public User(OS os) {
        this.os = os;
    }
    
    public abstract void login();
}

public class Developer extends User {
    private String id;
    private String pw;
    
    public Developer(OS os, String id, String pw) {
        super(os);
        this.id = id;
        this.pw = pw;
    }
    
    @Override
    public void login() {
        if(id.equals("") && pw.equals("")) {
            os.info();
        }
    }
}

public class Main {
    public static void main(String[] args){
        User user = new Developer(new MacOS(), "", "");
        user.login();
    }
}
  • OS 추상화 인터페이스
  • Linux, MacOS, Window : 인터페이스를 구체화한 객체
  • User : 기능 계층의 최상위 클래스
  • Developer : 기능 계층에서 새로운 부분을 확장한 클래스
  • OS(OS 객체)와 User(사용자 객체)를 분리함으로써 각자의 성질에만 책임을 가지면 된다. (SRP)

브릿지 패턴은 변경이 빈번한 클래스에 적합하지만, 응집도가 높아져 코드가 복잡해질 수 있다.
