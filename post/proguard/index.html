[android] Proguard(난독화)

0001-01-01
android

ProGuard는 Java 코드를 축소, 최적화 및 난독화하는 오픈 소스 명령줄 도구입니다.

  • 디컴파일시 코드가 노출되는 것을 방지
  • 불필요한 메소드를 제거하여 멀티덱스를 피할 수 있다.

멀티덱스
안드로이드 앱을 구성하는 코드는 컴파일 되어 덱스(dex) 파일로 만들어 집니다. 하나의 덱스(dex) 파일에는 최대 65536개의 메소드만 참조 할 수 있습니다. 만약 프로젝트의 코드가 65536개의 메소드를 초과하게 되면 덱스(dex)파일이 여러개가 생성됩니다. 그러면 멀티 덱스를 사용하여 컴파일 할 수 있지만, 빌드 과정에서 앱 내의 파일을 여러개의 덱스파일로 나누어야 하므로 빌드 속도가 느려지고 APK 용량이 커지게 됩니다.

        proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
    }
}

# classType : class, interface, enum
# methodType : 접근 지정자, return 타입 
# -keep [classType][packageName].[className] {
#    [methodType][methodName]([params]);
#}

# ex) com.daily.package의 하위 클래스를 전부 난독화하지 않음
-keep class com.daily.package.**

# ex) 패키지의 클래스중 public은 난독화 제외
-keep class com.daily.package.** {
    public <fields>;
    public <methods>;
}

# ex) com.daily.package 하위 클래스 중 public static만 난독화하지 않음
-keep class com.daily.package.** {    
    public static <fields>;    
    public static <methods>;    
    public <methods>;    
}

# 소스파일의 라인은 섞지 않음
-keepattributes SourceFile,LineNumberTable

# 소스파일의 변수명 변경
-renamesourcefileattribute SourceFile


# 모든 인터페이스를 난독화하지 않음
-keep interface * {
  <methods>;
  <fields>;
}

# 난독화 안하는 옵션
# -dontobfuscate 
# 최적화 안하는 옵션
# -dontoptimize

# 사용하지 않는 메소드 유지
-dontshrink 
# 경고 무시
-ignorewarnings
# 지정해서 무시
-dontwarn packName.**

# ex) retrofit 통신할 때는 model의 난독화를 없애 주어야 한다. 
-keep class packageName.model.** { *; }

# 라이브러리 jar를 읽는 동안 보이는 개인 또는 패키지 라이브러리 클래스를 간과하는 경향이 있으므로 아래 옵션을 사용하는 것이 좋다.
-dontskipnonpubliclibraryclasses
-dontskipnonpubliclibraryclassmembers

option 정리

  • -dontwarn : 경고 무시
  • -dontnote : Note를 무시한다 (proguard에서 Note는 컴파일러의 error)
  • -ignorewarnings : 경고 무시
  • -keep class|interface packageName.** : 난독화가 필요하지 않은 경우
  • -dontoptimize : 최적화 하지 않기
  • -dontshrink : 사용하지 않는 메소드 유지
  • -keepclassmembers : 특정 클래스 멤버 원상태 유지
  • -keepattributes : 내부 클래스 원상태 유지 적용
  • -dontskipnonpubliclibraryclassmembers : 패키지에 표시되는 라이브러리 클랙스 멤버를 무시하지 않도록 지정

프로가드를 사용한다면 라이브러리가 잘 작동되는지 확인해 봐야 하고, 구성이 잘못될 경우 애플리케이션이 충돌할 수 있습니다. 이러한 예상치 못한 결과는 부정적인 영향을 미칠 수 있습니다.
